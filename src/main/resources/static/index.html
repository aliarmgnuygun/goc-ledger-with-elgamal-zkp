Live Transfer — a real transaction on your own encrypted ledger

Three accounts (Alice=20, Bob=10, Carol=5) start with encrypted balances. Pick a sender, a receiver and an amount, then try an honest transfer or an attack.

Why only DL & EC here? The live ledger runs only with DL and EC. A transfer needs, on top of a range proof, a ciphertext-equivalence proof (new balance = current − amount). The Bulletproofs library we use only produces a standalone range proof and cannot output an equality proof inside the same ElGamal scheme — so it can't drive a transfer. The speed/size comparison of all three schemes lives in the benchmark charts.
Heads-up: a larger bit length is slower The bit length sets the range proof width (it proves the amount lies in [0, 2ⁿ)). EC stays fast at 32-bit (~0.3 s), but DL at 32-bit can take ~20 s per transfer (2048-bit modPow × 32 bits). For 32-bit we recommend EC — this is the live version of the "DL is the slowest" finding in the charts.

Reading the ciphertext Each balance is an ElGamal ciphertext of two group elements (c1, c2). The hex is truncated with …, but the byte count is real: an EC element is 32 bytes (Ristretto255 ≈ 256-bit group), a DL element is 256 bytes (ElGamal 2048-bit group). That ~8× size gap is exactly the EC-vs-DL trade-off.
